## 1. What You're Building

The Cow is a **special class** (class_id = 9, `is_special: true`). "Special" means it has **1 skin variant** instead of the standard 3. This affects frame counts (your sprites should already have 1/3 the frames of a normal class).

A class recolor is an **override mod**: you register class_id 9 with `"override": true`, and GNX replaces the vanilla Cow sprites with yours. The Cow keeps all her gameplay behavior. Only the visuals change.

**What this affects:** How the Cow looks on standard cells (WALL, RIDE, etc.), big cells (DAIRY, GIANT, CHAINS), and tent cells (BIND, T.WALL, etc.).

**What this does NOT affect:** The DAIRY cell's own background and milking-machine sprites (those are cell-level, not class-level). If you also want to recolor those, see [Optional: Recolor the DAIRY Cell](#optional-recolor-the-dairy-cell) at the end.

## 2. Prerequisites

**Software you need:**

- **GNX** installed via G3M (the mod manager). If you can launch the game and see `[GNX] loader start` in the debug log, you're good.
- **Python 3.9+** from [python.org](https://www.python.org/downloads/).
- **Pillow** (Python image library):
  ```
  pip install Pillow
  ```

**Tools folder:** All commands in this tutorial reference `<tools>/`, the folder containing the GNX modding scripts. If you got them from G3M, extract `tools.zip`. If from the repo, it's the `tools/` directory.

The tools you'll use:

| Script | What it does |
|--------|-------------|
| `gnx_pack_strips.py` | Packs sprite folders into strips. **Auto-fills frame counts** in your JSON. |
| `export_class_sprites.py` | Copies vanilla sprites from a UMT export and renames them for your mod. Generates the `sprites` JSON dict. |
| `build_mod.py` | All-in-one: pack + verify + deploy to the game folder. |
| `generate_class.py` | Interactive: asks you questions and writes a `classes.json` skeleton. |

**Debug log location:** `%LOCALAPPDATA%\goblin_nest\gnx_debug.txt` (Windows). You'll check this file after every launch.

## Step 1: Create the Mod Folder

Inside your game's install folder, find (or create) the `GNX_mods/` directory. Create your mod folder inside it:

```
<game folder>/
  GNX_mods/
    cow_recolor/           <-- create this
      manifest.json        <-- you'll create this next
      classes.json         <-- sprite mappings (Step 4)
      sprites/             <-- your recolored frame PNGs (Step 3)
      strips/              <-- auto-generated by the packer (Step 5)
```

**The folder name matters.** It becomes part of your mod's identity. Use lowercase, no spaces. In this tutorial we use `cow_recolor`.

---

## Step 2: Write manifest.json

Create `GNX_mods/cow_recolor/manifest.json`:

```json
{
  "mod_id": "cow_recolor",
  "name": "Cow Recolor",
  "version": "1.0.0",
  "compatible_game_versions": ["1.33"],
  "classes": "classes.json"
}
```

**Critical:** `"compatible_game_versions"` must include `"1.33"` (the current game version). If this doesn't match, GNX silently skips your entire mod.

**Checkpoint:** Launch the game now (before sprites or classes.json exist). Open `gnx_debug.txt`. You should see:

```
[GNX] loading: Cow Recolor v1.0.0
```

If you see `[GNX] version mismatch`, your `compatible_game_versions` is wrong. If you see nothing about your mod, your folder name doesn't match `mod_id`, or `manifest.json` has a syntax error.

**Stop here until this works.** If the mod isn't recognized now, nothing else will work.

## Step 3: Organize Your Sprites

You need to place your recolored PNGs into numbered folders inside `sprites/`. Each body part, for each animation phase, gets its own folder. Frames must be numbered starting from 0: `0.png`, `1.png`, `2.png`, etc.

### Understanding the Layer System

The game draws each body part in two passes:
- **spr_array** (base layer): the character outline, linework, and base colors.
- **spr_c_array** (color layer): the color overlay drawn on top.

Some parts use the **same** sprite for both layers (like the head). Others have separate `_c` variants. Your recolored sprites should follow the same pattern as vanilla.

### Frame Numbering

Each folder must contain numbered PNGs: `0.png`, `1.png`, `2.png`, etc.

Since you're recoloring, your frame count for each folder **must match** the vanilla Cow's frame count for that same body part. If you edited the vanilla frames directly, you already have the right count. If in doubt, count the PNGs in the vanilla source folder.

### Quick Sanity Check

After organizing, your `sprites/` folder should contain **54 subfolders** (2 shared + 16 standard + 15 big + 21 tent). If you're missing any, the game will show a fallback sprite or crash.

Run this in your mod's root to count:
```bash
dir /b sprites | find /c /v ""
```
Or on Linux/Mac:
```bash
ls sprites/ | wc -l
```

### Tool Shortcut: export_class_sprites.py

If you have a UMT sprite export of the vanilla game, `export_class_sprites.py` can copy and rename the vanilla cow sprites into your mod folder, giving you the exact frames to edit:

```bash
python <tools>/export_class_sprites.py --src cow --src-id 9 --dst cow_rc --mod-dir "GNX_mods/cow_recolor"
```

It also generates the `"sprites"` dict for your `classes.json`.

**Caveat:** The cow's standard-cell sprites use non-standard naming (`spr_h_base_idle_head_cow` instead of `spr_h_cow_idle_head`). The export script looks for `spr_h_cow_*` and will miss those. You'll need to copy the `spr_h_base_*_cow` sprites manually. Big and tent sprites (`spr_h_cow_big_*`, `spr_h_cow_tent_*`) are found normally.

### Shortcut: Let the Packer Count Frames for You

You don't have to count frames manually. Set every `"frames"` to `0` (or any placeholder), then run `gnx_pack_strips.py --force` (Step 5). The packer auto-detects how many PNGs are in each folder and **rewrites `"frames"` in your JSON** to the correct count. This is the recommended workflow: get the JSON structure right first, worry about frame counts never.

### What to Change in This JSON

**You MUST change (unless using the auto-fill shortcut above):**

- Every `"frames"` value: count the PNGs in each folder and put the real number. The values above are **estimates based on another is_special class** (Frieren). Your vanilla Cow sprites may have slightly different counts. **Count your actual files**, or let the packer do it.

**You CAN change (but don't need to for a basic recolor):**

- `"name"`: display name. Keep `"COW"` for a recolor.
- `"sprite_prefix"`: any unique prefix. We used `spr_h_cow_rc` (rc = recolor).
- Folder names: must match what's in `sprites/`. If you used different names, update the `"folder"` fields.

**Do NOT change:**

- `"class_id": 9` and `"override": true` (this is what makes it replace the vanilla Cow)
- `"is_special": true` (the Cow is a special class)
- `"has_hair": false` (Cow has no separate hair layer)
- `"hand_color": "gnx:hand_c"` (must point to your hand_c sprite key)
- The `"gnx:..."` references in the clothing maps (they point to keys in your `sprites` dict)
- Array slot positions (the `-1` and `"gnx:..."` order matters, each index is a specific body part)

### Understanding the Clothing Arrays

Each array index maps to a body part layer:

```
Index:  [0]    [1]    [2]      [3]    [4]   [5]        [6]
Part:   hair   head   breast   hand   leg   leg_part   cape/misc
```

`-1` means "no sprite for this slot." The cape (index 6) only exists in spr_array for standard cells. Big and tent cells don't use a cape.

### Understanding `"gnx:key"` References

`"gnx:idle_head"` tells GNX: "look up the key `idle_head` in the `sprites` dict above and use that sprite." This is how the clothing maps connect to your actual sprite files. The key after `gnx:` must exactly match a key in your `sprites` dict.

### Alternative: generate_class.py

If you prefer an interactive approach, `generate_class.py` asks you questions and generates a `classes.json` skeleton:

```bash
python <tools>/generate_class.py
```

It handles `is_special`, `override`, and `class_id` correctly. However, it generates the standard clothing format (`leg_1`/`leg_2` structs) instead of the `spr_array`/`spr_c_array` format that `is_special` classes require. **You will need to manually convert the clothing sections** to the array format shown above. For the Cow recolor, copy-pasting the template in this tutorial is faster.

## Step 5: Pack and Deploy

### Recommended: build_mod.py (all-in-one)

`build_mod.py` runs the packer, verifies all sprites are packed, and deploys the mod to the game folder in one command:

```bash
# Preview (no changes):
python <tools>/build_mod.py "GNX_mods/cow_recolor" "<game folder>" --dry-run --force

# Build and deploy:
python <tools>/build_mod.py "GNX_mods/cow_recolor" "<game folder>" --force
```

Replace `<game folder>` with the path to the folder containing `GoblinNest.exe`. The script creates/replaces `<game folder>/GNX_mods/cow_recolor/` with a clean copy containing only `manifest.json`, `classes.json`, and `strips/`.

### Manual alternative: gnx_pack_strips.py

If you prefer to pack and deploy separately:

```bash
# Preview first (no files changed):
python <tools>/gnx_pack_strips.py "GNX_mods/cow_recolor" --dry-run

# Pack for real:
python <tools>/gnx_pack_strips.py "GNX_mods/cow_recolor" --force
```

Then copy `manifest.json`, `classes.json`, and `strips/` to `<game folder>/GNX_mods/cow_recolor/` yourself.

`--force` repacks everything even if strips already exist. Use it the first time, and whenever you change sprites.

**What happens:**

1. The packer creates a `strips/` folder inside your mod.
2. Each sprite folder becomes a single `.png` strip (all frames side by side horizontally).
3. `classes.json` is rewritten in-place: each sprite entry gets a `"strip"` key added, like:
   ```json
   "idle_head": {
     "strip": "strips/idle_head.png",
     "frames": 30,
     "xorig": 0,
     "yorig": 90,
     "folder": "spr_h_cow_rc_idle_head"
   }
   ```
4. A backup `classes.json.bak` is created.

**If the packer reports errors:**

- `missing frames`: the folder name in `"folder"` doesn't match an actual folder in `sprites/`.
- `0 frames found`: the folder exists but contains no PNGs, or PNGs aren't numbered correctly.

**After packing, your mod folder should look like:**

```
cow_recolor/
  manifest.json
  classes.json        <-- now has "strip" keys
  classes.json.bak    <-- backup
  sprites/            <-- your source PNGs (can delete after packing to save space)
  strips/             <-- packed strips (this is what the game loads)
```

## Step 6: Test In-Game

1. Launch the game.
2. Open `%LOCALAPPDATA%\goblin_nest\gnx_debug.txt`.

**What success looks like:**

```
[GNX] loading: Cow Recolor v1.0.0
[GNX] class_id 9 override
[GNX] idle_head: strip frames=30 idx=ref sprite 1530
[GNX] idle_breast: strip frames=30 idx=ref sprite 1531
... (one line per sprite)
[GNX-TEST] 35/35 passed
```

Key things to check:

- `class_id 9 override` confirms the Cow class is being replaced.
- Each sprite line should show `strip frames=N` with the correct frame count.
- `35/35 passed` at the end means the test suite found no issues.

**If something is wrong:**

- `[GNX] version mismatch: mod=cow_recolor game=1.33`: your `compatible_game_versions` is wrong.
- `[GNX] sprite not found: strips/...`: a strip file is missing. Re-run the packer.
- `[GNX-TEST] FAIL T22`: a clothing map references a sprite that doesn't resolve. Check that every `"gnx:key"` in the clothing sections matches an actual key in `sprites`.
- No mention of your mod at all: `manifest.json` has a syntax error or `mod_id` doesn't match the folder name.

**In-game visual test:**

1. Start or load a save that has a Cow unit.
2. Place her in a standard cell (any WALL). Check that your recolored sprites show correctly.
3. Place her in a big cell (DAIRY, GIANT). Check big cell sprites.
4. Place her in a tent cell (BIND, T.WALL). Check tent sprites including the birth animation.

If a body part shows the vanilla sprite instead of yours, that specific `"gnx:key"` reference is broken. If a body part is invisible, the strip is missing or has 0 frames. If the animation stutters or desyncs, the frame count in `classes.json` doesn't match the actual number of frames in the strip.

A class override does NOT change DAIRY sprites. If you also want to recolor the milking scene, you need a `cells.json` that patches h_type 36.

**This is optional.** Skip this section if you only want to recolor the Cow on standard/big/tent cells.

## FAQ and Troubleshooting

### "The game crashes on launch"

Open `gnx_debug.txt` and look for the last line before it stops. Common causes:

- **Missing sprite strip**: a `"strip"` path in classes.json points to a file that doesn't exist. Re-run the packer.
- **Frame count mismatch**: `"frames"` in JSON says 30 but the strip only has 20 frames. The animation system reads past the end.
- **JSON syntax error**: a missing comma, bracket, or quote. Paste your JSON into [jsonlint.com](https://jsonlint.com/) to validate.

### "My mod loads but the Cow looks the same as vanilla"

- Check that `"override": true` is present in classes.json.
- Check that `"class_id": 9` is present.
- If the log says `class_id 9 collision -- skipped` instead of `class_id 9 override`, you forgot `"override": true`.

### "Some body parts are invisible"

- The `"gnx:key"` in the clothing map doesn't match any key in `sprites`. Check spelling.
- The sprite strip has 0 frames (empty folder or wrong folder name).

### "The animation is desynced or stuttering"

Your `"frames"` count doesn't match the actual number of frames in the strip. Count the PNGs in the source folder and update the JSON.

### "I only want to recolor the _c (color) sprites, not the base"

You still need to provide ALL sprites (both base and color). A class override replaces the entire class entry. You can keep the base sprites identical to vanilla and only change the `_c` variants, but all folders must exist.

### "My sprites look offset or misaligned"

Check `"xorig"` and `"yorig"`. For the Cow:

- Hand: `3, 1`
- Everything else: `0, 90`

If your frames have a different canvas size than vanilla, the origin might need adjusting. `yorig` should equal the frame height (typically 90px for body parts).

### "I see [GNX-TEST] FAIL T22"

T22 checks that every sprite referenced in clothing maps resolves to a valid sprite. The error message tells you exactly which key failed, like `c9.clothing_big.phase_0.spr_array[1]`. Cross-reference with your `sprites` dict.

### "Do I need to recolor the goblin sprites too?"

No. Goblin sprites (the monster on top of the Cow) are cell-level, not class-level. A class recolor doesn't affect them.

### "What about the ogre carry sprite?"

`spr_ogre_carry_base_cow` (the sprite when an ogre carries the Cow) is hardcoded in vanilla. A basic class override doesn't change it. If you want to override it, add a `"mon_spr_overrides"` section to your class entry:

```json
"mon_spr_overrides": {
  "patrol": "gnx:carry_head"
}
```

And add a `"carry_head"` entry to your `sprites` dict with the ogre carry sprite.

### "What about the G.BANG 1 breast sprite?"

Same situation. Add `"gb1_breast_d2": "gnx:gb1_breast"` to your class entry and provide the sprite.

### "Can two mods both override the Cow?"

No. The last one to load wins (alphabetical folder order). Only one override per class_id.

### "How do I revert to vanilla?"

Delete your mod folder from `GNX_mods/` (or rename `manifest.json` to something else). Relaunch. The vanilla Cow is back. No save file changes needed.

### "frame count reference: how many frames does each vanilla Cow sprite have?"

Since the Cow is `is_special: true` (1 skin variant), expect these approximate ranges:

| Cell Type | Phase | Head | Breast | Leg | Leg Part | Cape |
|-----------|-------|------|--------|-----|----------|------|
| Standard | idle | 30 | 30 | 30 | 30 | 30 |
| Standard | loop | 75 | 75 | 75 | 75 | 75 |
| Big | start | ~12 | ~12-24 | ~12-24 | n/a | n/a |
| Big | idle | ~16 | ~16-32 | ~16-32 | n/a | n/a |
| Big | loop | ~35 | ~35-70 | ~35-70 | n/a | n/a |
| Tent | idle | ~14 | ~14-28 | ~14-56 | ~14-28 | n/a |
| Tent | loop | ~35 | ~35-70 | ~35-140 | ~35-70 | n/a |
| Tent | birth | ~14 | ~14-54 | ~14-56 | ~14-28 | n/a |

**These are estimates.** Always count the actual frames in your source folders.

The breast and leg sometimes have double frames (base has 2x the head's count, _c variant matches the head's count). This is because the base sprite encodes a 2-frame sub-animation that the color overlay doesn't need. Check your vanilla source files.

### "Can I do a partial recolor (only standard cells, skip big/tent)?"

No. An override replaces the entire class entry. If you omit `clothing_big`, the Cow won't render on big cells at all. You must provide all three clothing sections.

### "I want to make a NEW class based on the Cow, not replace her"

Change your classes.json to:

```json
{
  "name": "MY_COW",
  "override": false,
  "is_special": true,
  ...
}
```

Remove the `"class_id": 9` line entirely. GNX will auto-assign a new ID. Your class will appear alongside the vanilla Cow, not replace her. You'll also need `"raid_spawns"` to make her appear in encounters.
